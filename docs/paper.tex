\begin{abstract}
Distributed systems that serve real-time analytics need to spread incoming requests across multiple backend workers without creating hotspots or slowing down responses. In this paper, we build a containerized testbed with three worker services and a custom load balancer that supports three routing algorithms: static round-robin, least-connections, and a latency-aware strategy based on Exponentially Weighted Moving Average (EWMA) scoring. We compare these algorithms under four scenarios: constant load, traffic spikes, gradual stress increase, and worker failure injection. Our results show that the latency-aware algorithm gives the best trade-off between low tail latency and fair distribution, especially when workers have different processing speeds or when one of them crashes. The least-connections algorithm performs well under uniform conditions but struggles with heterogeneous workers. Round-robin is simple and predictable but cannot adapt to changing conditions.
\end{abstract}

\section{Introduction}

When you have a web application that gets a lot of traffic, you cannot run everything on a single server. You need multiple backend servers (we call them ``workers'') to share the load. The component that decides which worker gets each request is called a \textit{load balancer}.

The simplest approach is round-robin: send the first request to worker 1, the second to worker 2, the third to worker 3, then go back to worker 1, and so on. This works fine when all workers are identical and equally fast. But in reality, workers might have different hardware, some might be slower due to garbage collection pauses, or one might crash entirely.

Dynamic load balancing means the system watches what is happening in real time (how fast each worker responds, how many requests are queued) and uses that information to make smarter routing decisions.

Our research question is: \textbf{Which dynamic load-balancing strategy provides the best trade-off between responsiveness, stability, and overhead in a distributed real-time analytics dashboard under variable workload, heterogeneous workers, and failures?}

To answer this, we built a containerized distributed system from scratch and ran controlled experiments comparing three algorithms. This paper presents the system design, the experiment methodology, and the results.

\section{Related Work}

Load balancing is a well-studied problem. NGINX \cite{nginx} and HAProxy \cite{haproxy} are two widely-used production load balancers that support many algorithms. Kubernetes \cite{kubernetes} has its own built-in load balancing through Services.

The classic algorithms include:
\begin{itemize}
    \item \textbf{Round-robin}: cycles through workers in order. Simple but blind to actual load \cite{tanenbaum}.
    \item \textbf{Least-connections}: picks the worker handling the fewest requests right now. Better under uneven loads \cite{cardellini}.
    \item \textbf{Weighted strategies}: assign weights based on worker capacity or measured performance \cite{pai1998}.
    \item \textbf{Latency-based}: use response time measurements to prefer faster workers. Envoy Proxy \cite{envoy} uses this approach.
\end{itemize}

Our latency-aware algorithm is inspired by Envoy's approach of using EWMA (Exponentially Weighted Moving Average) to smooth out latency measurements, combined with a queue-depth penalty similar to what is described in \cite{mitzenmacher2001}.

The key difference in our project is that we focus on a specific use case (real-time analytics) and we evaluate under failure conditions, which many benchmark studies skip.

\section{System Design}

\subsection{Architecture Overview}

Our system has four main components:

\begin{enumerate}
    \item \textbf{Workers} (3 instances): FastAPI services that simulate analytics event processing. Each worker can introduce configurable delay (simulating I/O) and CPU work.
    \item \textbf{Load Balancer}: A custom FastAPI reverse proxy that intercepts all client traffic and forwards it to a selected worker.
    \item \textbf{Monitoring}: Prometheus scrapes metrics from the load balancer every 2 seconds. Grafana provides dashboards.
    \item \textbf{Load Generator}: k6 generates synthetic HTTP traffic with configurable virtual users, duration, and request parameters.
\end{enumerate}

Everything runs in Docker containers orchestrated by Docker Compose.

\subsection{Worker Service}

Each worker exposes:
\begin{itemize}
    \item \texttt{/api/v1/health} -- health check endpoint
    \item \texttt{/api/v1/analytics/event} -- processes an analytics event with configurable delay and CPU usage
    \item \texttt{/api/v1/crash} -- kills the process (for failure testing)
\end{itemize}

Workers accept parameters \texttt{delay\_ms} and \texttt{cpu\_ms} which control how long each request takes. A \texttt{SLOWDOWN\_FACTOR} environment variable multiplies these values, allowing us to simulate heterogeneous hardware (e.g., worker-3 runs at 2.5x slowdown).

\subsection{Load Balancer Algorithms}

\subsubsection{Round-Robin (Baseline)}
Cycles through healthy workers in order. Uses a shared counter with an async lock to ensure thread safety. When a worker becomes unhealthy, it is removed from the rotation.

\subsubsection{Least-Connections}
Selects the worker with the fewest in-flight (currently active) requests. Ties are broken by EWMA latency. This is adaptive because it naturally avoids overloading a slow worker since slow workers finish requests later and accumulate more in-flight connections.

\subsubsection{Latency-Aware (EWMA + Queue Penalty)}
Computes a score for each worker:

\begin{equation}
    \text{score}(w) = \text{ewma\_latency}(w) + 20 \times \text{in\_flight}(w) + 5 \times \text{failed}(w)
\end{equation}

The worker with the lowest score gets the request. The EWMA latency is updated after every request and every health check using:

\begin{equation}
    \text{ewma}_{new} = \alpha \times \text{measured} + (1 - \alpha) \times \text{ewma}_{old}
\end{equation}

where $\alpha = 0.25$. This gives more weight to recent measurements while not overreacting to a single slow request.

\subsection{Health Checking}

The load balancer runs a background health check loop that pings every worker every second. If a worker fails to respond or returns a 5xx status, it is marked unhealthy and removed from the candidate pool. When it recovers, it automatically re-enters the pool.

\subsection{Metrics and Observability}

The load balancer exposes Prometheus-compatible metrics at \texttt{/lb/metrics}:
\begin{itemize}
    \item Total and failed request counts
    \item Latency sum, p95, and p99
    \item Per-worker: health status, in-flight count, EWMA latency, score, request count
\end{itemize}

\section{Discussion}

\subsection{Key Findings}

\begin{enumerate}
    \item The latency-aware algorithm consistently provides the lowest tail latency (p95/p99) across all scenarios because it accounts for both response time and queue depth.
    \item Least-connections works well when workers have similar speeds but underperforms with heterogeneous workers because it only looks at connection count, not actual speed.
    \item Round-robin is the simplest and has the lowest overhead (no scoring computation) but cannot adapt to any runtime conditions.
\end{enumerate}

\subsection{Trade-offs}

\begin{itemize}
    \item \textbf{Overhead}: Round-robin has O(1) selection. Least-connections needs O(n) to find the minimum. Latency-aware also O(n) but with more computation per worker. With only 3 workers this is negligible, but at scale it matters.
    \item \textbf{Stability}: The EWMA smoothing in latency-aware prevents oscillation (rapidly switching between workers). Without smoothing, a single slow request could cause the algorithm to avoid a healthy worker.
    \item \textbf{Fairness}: Round-robin is perfectly fair (equal distribution). Adaptive algorithms intentionally create unfair distribution to improve overall performance.
\end{itemize}

\subsection{Limitations}

\begin{itemize}
    \item Our testbed uses only 3 workers. In production, load balancers handle hundreds or thousands of backends.
    \item The workload is synthetic (configurable delay + CPU). Real analytics workloads have more complex patterns.
    \item We run everything on a single Docker host, so network latency between components is negligible. In a real distributed system, network jitter would be a factor.
    \item The EWMA alpha value (0.25) was chosen empirically. A proper study would need to test multiple values.
    \item Our experiments run for 60 seconds. Long-running production workloads might exhibit different patterns.
\end{itemize}

\subsection{Safety, Privacy, and Deployment Notes}

\begin{itemize}
    \item \textbf{Failure assumptions}: We assume fail-stop behavior (a crashed worker stops responding). Byzantine failures (worker returns wrong data) are not handled.
    \item \textbf{Attack surface}: The load balancer exposes management endpoints (\texttt{/lb/stats}, \texttt{/lb/workers}) that reveal internal state. In production, these should be access-controlled.
    \item \textbf{Operational safety}: The crash endpoint (\texttt{/api/v1/crash}) exists only for testing. In production, failure injection should be behind authentication.
    \item \textbf{Reproducibility}: All experiments are scripted and containerized. The same results can be obtained on any machine with Docker installed, though absolute numbers will differ.
    \item \textbf{Not production-ready}: This is a course project demonstrating concepts. A production load balancer needs TLS termination, connection pooling, circuit breakers, retries, and much more.
\end{itemize}

\section{Conclusion}

We built a containerized distributed system testbed to compare three load-balancing algorithms: round-robin, least-connections, and latency-aware routing. Through four experiment scenarios (constant load, traffic spikes, stress testing, and failure injection), we showed that the latency-aware algorithm with EWMA scoring provides the best trade-off between response time and stability, especially in conditions where workers are not uniform or when failures happen.

The main takeaway is that a load balancer that uses real-time feedback (actual response times) makes better decisions than one that only counts connections or blindly rotates. The cost is slightly more complexity in the scoring logic, but with modern hardware this overhead is negligible for small to medium clusters.

Future work could explore reinforcement-learning-based scheduling, predictive autoscaling, or cost-aware placement in cloud environments.
